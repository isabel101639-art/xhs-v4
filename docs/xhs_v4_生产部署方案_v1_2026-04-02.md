8. 编排方式：`Docker Compose`

## 3. 运行方式

在生产机上进入项目目录后，推荐使用：

```bash
docker compose up -d --build
```

查看运行状态：

```bash
docker compose ps
docker compose logs -f web
docker compose logs -f worker
docker compose logs -f nginx
```

健康检查地址：

```text
http://<your-host>/healthz
```

## 4. 环境变量

核心环境变量包括：

1. `SECRET_KEY`
2. `DEEPSEEK_API_KEY`
3. `POSTGRES_DB`
4. `POSTGRES_USER`
5. `POSTGRES_PASSWORD`
6. `DEFAULT_TOPIC_QUOTA`
7. `SESSION_COOKIE_SECURE`
8. `GUNICORN_WORKERS`
9. `GUNICORN_THREADS`

## 5. 当前配置特性

当前应用已经支持：

1. 通过 `DATABASE_URL` 切换数据库
2. 通过 `SECRET_KEY` 切换会话密钥
3. 通过 `DEFAULT_TOPIC_QUOTA` 控制默认名额
4. 通过 `DEEPSEEK_API_URL` 切换 AI 接口地址
5. 通过 `healthz` 做健康检查

## 6. 数据库说明

### 开发环境

当前本地仍可继续使用：

1. `SQLite`

### 生产环境

建议使用：

1. `PostgreSQL`

### 注意事项

当前项目已经具备自动建表和基础字段迁移逻辑，但：

1. `SQLite -> PostgreSQL` 的历史数据迁移脚本还没有单独写。
2. 如果你要把现有本地数据带到正式环境，后续还需要补一版迁移方案。

## 7. 生产建议

1. 正式环境不要继续直接使用本地 `.env` 中的真实密钥。
2. 正式环境应通过服务器环境变量或受控的部署变量注入密钥。
3. `SESSION_COOKIE_SECURE` 在 HTTPS 环境下建议改为 `true`。
4. PostgreSQL 和 Redis 建议挂载独立数据卷，并做定期备份。
5. Nginx 前面如果再接云负载均衡，也建议保留 `healthz` 探针。
