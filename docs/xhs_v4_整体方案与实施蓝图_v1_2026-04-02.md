## 1. 方案结论

`xhs_v4` 建议采用“基于现有 Flask 单体逐步升级”的路线，不建议推倒重做。

推荐建设方向：

1. 以现有 `Flask + SQLAlchemy + Jinja + Bootstrap` 为基础继续迭代。
2. 先完成 `P0 可运营版本`，保证业务闭环先跑起来。
3. 热点、语料、账号数据、文案、图片等自动化能力统一纳入后台中台。
4. 强制建立 `储备池 -> 候选池 -> 正式池 -> 归档池` 的内容隔离机制。
5. 对外部数据源采用 `官方接口 + 第三方平台 + 人工导入兜底` 的混合模式。

## 2. 产品形态

## 3. 推荐技术路线

## 3.1 开发策略

推荐采用两段式演进：

### 阶段 A：单体重构增强

保留现有主工程，按模块拆分代码结构：

1. `routes/`
2. `services/`
3. `models/`
4. `jobs/`
5. `repositories/`
6. `templates/`
7. `static/`
8. `docs/`

### 阶段 B：后台能力独立化

当热点抓取、AI 图片生成、账号同步任务稳定后，再逐步将异步任务、数据同步和重计算能力拆分出去。

## 3.2 推荐技术栈

### 当前沿用

1. 后端：`Flask`
2. ORM：`SQLAlchemy`
3. 模板：`Jinja2`
4. 前端样式：`Bootstrap`

### 建议新增

1. 数据库：`PostgreSQL`
   说明：开发期可继续用 SQLite，本地验证快；正式环境建议切 PostgreSQL。
2. 缓存/队列：`Redis`
3. 定时与异步任务：`Celery` 或 `RQ` + `APScheduler`
4. 文件存储：本地目录起步，后续可切对象存储
5. 表单与接口校验：`Pydantic` 或 `Marshmallow`
6. 管理配置：`.env + admin settings`

## 3.3 为什么不建议推倒重来

原因：

1. 当前系统已有大量业务规则沉淀在 `app.py` 中。
2. 话题、提报、分析、自动化的核心逻辑已基本成型。
3. 直接重写会让业务规则丢失，且验证成本更高。
4. 先把结构拆干净，再升级数据库和任务系统，风险更低。

## 4. 外部数据源接入策略

## 4.1 总原则

所有外部数据能力统一走三层模式：

1. `官方接口优先`
2. `第三方平台补充`
3. `人工导入兜底`

## 4.2 平台接入建议

| 场景 | 抖音 | 小红书 | 建议 |
|---|---|---|---|
| 热点词/热榜 | 可优先走官方 | 不建议默认按官方公开能力规划 | 抖音官方 + 小红书第三方 |
| 创作者账号表现 | 可授权接入 | 不建议默认按官方直连接口规划 | 抖音授权 + 小红书第三方 |
| 爆款笔记/内容洞察 | 官方能力有限 | 官方公开能力不适合直接规划 | 第三方平台为主 |
| 兜底方案 | Excel/CSV 导入 | Excel/CSV 导入 | 两边都保留 |

## 5. 系统整体架构

## 5.1 逻辑分层

建议分为六层：

1. `展示层`
   前台网站、后台管理台、数据看板
2. `接口层`
   Web 页面路由、JSON API
3. `业务层`
   期数管理、热点抓取、语料处理、话题生成、审核发布、账号分析
4. `任务层`
   定时任务、同步任务、AI 生成任务、归档任务
5. `数据层`
   PostgreSQL / SQLite、缓存、文件存储
6. `外部接入层`
   抖音开放平台、千瓜、飞瓜、AI 模型服务

## 5.2 内容池架构

内容必须按池子隔离：

1. `储备池`
   热点、语料、模板、图库原始资产
2. `候选池`
   自动生成但未审核的话题、文案、图片
3. `正式池`
   已发布到当前期的话题、图片、模板快照
4. `归档池`
   已结束期数的完整快照

## 5.3 活动期快照架构

每期发布时生成快照：

1. 话题快照
2. 语料快照
3. 图片快照
4. 文案模板快照
5. 数据口径快照
6. 报表快照

2. `科普问题池`
   关键词覆盖：`脂肪肝怎么吃 / 脂肪肝做什么运动 / 如何减肥 / 肥胖怎么减 / 养肝护肝 / 保肝护肝`
   目标：发现“怎么做”“怎么吃”“怎么运动”“怎么减肥”这类可做成收藏型内容的问题
3. `平台热搜/蹭热点池`
   关键词覆盖：`养肝护肝 / 减肥 / 肥胖 / 体检 / 中医养生 / 健康养生`
   目标：发现适合蹭平台流量的热搜热榜内容

### 时间窗口补充

自动化搜索默认支持：

1. `近3天`
2. `近7天`
3. `近30天`
4. `自定义日期`

### 自动化发现后的归宿

自动搜索出来的内容统一先进发现池，再经人工确认分流到不同归宿：

1. `当前话题广场`
   用途：适合本期直接执行的内容
2. `下一期储备`
   用途：适合进入下一期候选池和储备内容
3. `肝健康IP栏目`
   用途：适合长期科普、栏目沉淀、模板复用
4. `热搜话题`
   用途：适合蹭热点，但需要同时给出融合建议和执行建议

### 自动化确认台补充

热点发现池不再只是原始表格，而是要补齐“规划判断 + 推荐归宿 + 一键执行”三层能力：

1. 每条热点自动计算 `规划分`
   说明：综合热度、高搜索问题特征、肝健康相关度、执行价值，给出更贴近运营判断的优先级
2. 每条热点自动给出 `推荐归宿`
   可选：`发当前话题广场 / 放下一期储备 / 放肝健康IP栏目 / 放热搜话题`
3. 每条热点自动给出 `推荐理由`
   说明：明确告诉运营“为什么更适合放这里”，不是只给一个按钮
4. 运营确认后支持：
   - 单条 `按建议分流`
   - 当前筛选结果 `批量按建议分流`
5. 如果推荐归宿是 `下一期储备`
   实际落点应为 `TopicIdea 候选话题池`，后续仍需人工审核后才能发布到正式活动
6. `TopicIdea 候选话题池` 不应只保留裸标题
   应默认带上：
   - 来源热点数量 / 参考爆款链接
   - 策划判断（角度、人设、软植入建议）
   - 执行建议（封面建议、配图建议、合规提醒）
   - 文案提示词

### 热搜话题页的定位

`热搜话题` 页面不是纯资讯页，而是“热点作战台”，每条内容应包含：

1. 热点/问题本身
2. 为什么值得蹭
3. 能切到哪个肝健康方向
4. 适合什么人设写
5. 怎么融合成我们的内容
6. 下一步可执行动作：
   - 发到当前期
   - 放入下一期
   - 放入肝健康IP栏目
   - 直接生成文案/图片
7. 页面应直接给出：
   - 标题方向
   - 开头钩子
   - 封面打法
   - `保存为热点预设`
   - `保存并去生成文案`
8. 热搜预设应同时带入：
   - 文案方向 / 标题技能包 / 图片技能包
   - 图片主类型
   - 工作流模式
   - 封面样式 / 内页样式
9. 图片模板 Agent 的对外表达应优先使用用户能理解的模板分类，而不是内部 key：
   - `大字封面`
   - `清单卡`
   - `课堂笔记卡`
   - `报告解读卡`
   - `备忘录/陪伴卡`
10. 这层推荐不应只在 `热搜话题` 页出现，也应同步出现在 `自动化中心` 的热点确认台，方便运营在后台就完成“看热点 + 判断图怎么做 + 确认分流”。
11. `自动化中心` 里的热点内容应支持：
   - 按图片模板 Agent 自动分桶
   - 一键预填图片方案
   - 先预览图片计划，再决定是否绑定报名记录发起真实图片任务
12. `候选话题池` 也应支持同样的图片模板分桶和一键预填，避免热点进入下一期后图文链路断开。
13. 图片模板 Agent 下一阶段不直接批量发图，而是先支持“按桶批量生成图片草案预览”，由运营确认后再决定哪些要进入真实图片任务。
14. 草案工作台中的单条草案应支持：
   - `保留`
   - `忽略`
   - `套用到图片任务表单`
   目的：先完成人审和微调，再决定是否真的发起图片任务。
15. 已保存的图片草案应支持：
   - 单条转为 `待执行图片任务`
   - 批量转为 `待执行图片任务`
   - 转换后默认不自动跑图，只进入任务中心等待后续补报名ID或进一步确认
16. `待执行图片任务` 还应支持：
   - 手工补报名ID
   - 补完后再正式派发图片任务
   这样图片自动化既能保持闭环，又不会绕过人工确认。
17. `自动化中心` 顶部应展示全链路漏斗总览：
    - 热点数
    - 候选话题数
    - 图片草案数
    - 待执行图片任务数
    - 已完成图片任务数
    让运营能一眼判断当前卡在哪一层。
18. 后台应提供单独的“本周交付清单页”，集中展示：
    - 本周完成定义
    - 三大板块完成度
    - 外部依赖
    - 上线前检查单
    用来做本周验收，而不是让运营在多个页面之间来回找状态。

### Agent 衔接补充

`肝健康IP Agent` 的规划结果需要能继续影响：

1. 文案生成方向
2. 标题技能包
3. 图片技能包
4. 热搜内容融合策略
5. IP栏目规划排序

原则：所有自动发现内容默认先进候选区，不自动直接发布，必须经过人工确认。

## 14. 建议的开发顺序

推荐开发顺序如下：

1. 先整理工程结构，拆分 `app.py`
2. 先做 `活动期隔离 + 正式池/储备池`
3. 再做 `热点池 + 语料库 + 候选话题池`
4. 再做 `审核发布流`
5. 再做 `AI 文案中心升级`
6. 再做 `全局数据看板升级`
7. 再做 `图片库`
8. 最后做 `账号数据看板`

## 15. 当前开发建议

进入正式开发前，建议先固定以下默认决策：

1. 开发路线：`沿用 Flask，渐进式重构`
2. 数据库路线：`开发期 SQLite，目标 PostgreSQL`
3. 抖音热点：`官方`
4. 抖音账号数据：`官方授权`
5. 小红书热点与账号数据：`千瓜`
6. 第三方不可用时：`人工导入兜底`
7. 每月话题产出：`80`
8. 每话题名额：`按期数/批次/单话题可调整，系统保留默认值`
9. 内容池模型：`储备池 -> 候选池 -> 正式池 -> 归档池`
10. 审核原则：`自动生成内容默认不直接发布`

## 16. 最终判断

该项目已经具备进入正式开发阶段的条件。  
当前最适合的方式不是继续零散改功能，而是按本方案进入模块化实施。

开发上建议先完成 `P0 可运营版本`，待你确认方案后，我们即可按模块开始落地。
